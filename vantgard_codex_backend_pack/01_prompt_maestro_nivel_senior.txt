PROMPT MAESTRO — VANTGARD BACKEND-FIRST ORCHESTRATOR (NIVEL SENIOR)

Actúa como un Principal Software Engineer, Django Backend Architect, Tech Lead y revisor de pull requests trabajando dentro de un repositorio real existente.

MISIÓN
Trabaja sobre el proyecto actual "Vantgard" para convertirlo en un e-commerce de licores premium con una base backend seria, mantenible y preparada para crecer. Tu foco principal es el backend, la lógica de negocio, la integridad de datos, la seguridad, la trazabilidad de pedidos y la arquitectura para pagos y seguimiento. El frontend es secundario y solo debe ajustarse cuando sea necesario para soportar el flujo funcional.

MODO DE TRABAJO
- No recrees el proyecto desde cero.
- Inspecciona primero la estructura real del repo y reutiliza lo existente.
- Trabaja en una rama dedicada.
- Realiza cambios incrementales, coherentes y revisables.
- Agrupa el trabajo en commits o lotes claros.
- Cuando termines, crea un pull request con resumen técnico, riesgos, dependencias, migraciones y pasos de prueba.

REGLAS NO NEGOCIABLES
1. No renombrar apps, paquetes, rutas ni templates existentes salvo necesidad técnica real.
2. No eliminar funcionalidades existentes sin justificarlo.
3. No introducir secretos reales, credenciales ni claves privadas.
4. Si una integración externa requiere credenciales, dejar variables de entorno y documentación clara.
5. No priorizar maquillaje visual por encima de lógica, seguridad y mantenibilidad.
6. Si algo es demasiado grande para una sola iteración, entregar un MVP backend sólido y dejar la siguiente fase documentada.
7. Todo cambio de modelo debe venir acompañado de migraciones y revisión de impacto.
8. Todo flujo crítico debe tener validaciones.
9. Aprovecha el admin de Django como herramienta real de operación.
10. Antes de finalizar, revisa URLs, imports, formularios, settings, admin y compatibilidad general.

OBJETIVOS TÉCNICOS PRIORITARIOS

FASE A — INSPECCIÓN Y BASE TÉCNICA
- Inspecciona la estructura actual del proyecto.
- Resume qué ya existe en modelos, vistas, formularios, admin, URLs y settings.
- Detecta deuda técnica, faltantes y piezas inconsistentes.
- Confirma si ya existe CustomUser, Pedido, Producto, Blog, GiftExperience, etc.
- No improvises una arquitectura paralela si ya existe una base útil.

FASE B — DOMINIO DE PRODUCTOS Y FILTROS REALES
Implementa o endurece:
- categorías de producto sin solapamiento ambiguo
- subcategorías coherentes si hace falta
- filtros backend por:
  - categoría única
  - rango de precio
  - ofertas activas
  - edición limitada
  - disponibilidad/stock
- consultas eficientes y preparadas para crecimiento

FASE C — CARRITO Y CHECKOUT SERIOS
Implementa una base funcional real para:
- carrito persistente (sesión o base de datos, según convenga)
- agregar producto
- actualizar cantidad
- eliminar producto
- vaciar carrito
- resumen de subtotal, descuentos, costo de entrega y total
- flujo de checkout coherente
- validaciones de stock antes de confirmar compra
- protección frente a cantidades inválidas o stock insuficiente

FASE D — ENTREGA Y LOGÍSTICA
Implementa:
- selector de tipo de entrega:
  - retiro en local
  - despacho local
  - express si aplica
- validación de campos de despacho
- soporte para sucursal si el retiro es presencial
- estructura de cálculo de costo de envío desacoplada
- lógica preparada para reglas de Antofagasta y extensible a otras comunas

FASE E — PAGOS BACKEND-FIRST
NO integres producción real si no hay credenciales.
Sí debes dejar:
- una capa de abstracción de pagos
- modelos o estructuras para intento de pago / referencia / estado de pago
- flujo de checkout preparado para:
  - transferencia bancaria
  - tarjeta crédito/débito (modo sandbox o mock seguro)
- estados claros:
  - pending
  - authorized
  - paid
  - failed
  - refunded
- puntos de integración limpios
- documentación de qué variable o servicio faltaría para producción

FASE F — TRACKING Y TOKEN
Implementa:
- token único legible por humanos para pedido, por ejemplo VNT-2026-XXXXXX
- endpoint o vista de seguimiento por token
- transición de estados:
  - recibido
  - preparando
  - listo para retiro
  - en camino
  - entregado
  - cancelado
- lógica coherente entre tipo de entrega y estados disponibles
- administración simple del estado desde Django admin
- si el QR es viable, generarlo para el token o comprobante del pedido

FASE G — QR AUTOMÁTICO
Implementa una primera versión robusta:
- QR generado al confirmar el pedido o al entrar a estado pagado
- almacenar la imagen de forma compatible con Django media
- enlazarla al pedido
- documentar dependencia usada
- no hacer hacks frágiles con rutas absolutas

FASE H — OPERACIÓN ADMINISTRATIVA
Mejora el admin para operación real:
- búsqueda útil
- filtros por estado
- stock crítico
- campos readonly cuando corresponda
- acciones administrativas si aportan valor
- visualización rápida de pedidos activos
- claridad para revisar pagos, tracking y datos de despacho

FASE I — API / SOMMELIER CONSULTING
Solo si las fases anteriores quedan sólidas:
- crea una capa de servicio desacoplada para integrar una API externa de licores
- nunca acoples la app directamente a una API externa desde templates
- maneja errores, timeouts y respuestas faltantes
- deja fallback seguro cuando la API no esté disponible

CRITERIOS DE CALIDAD
- Código legible, modular y mantenible
- Separación entre dominio, formularios, vistas, servicios y utilidades cuando aporte valor
- Evita meter lógica pesada directamente en templates
- Evita meter lógica compleja directamente en admin si puede vivir mejor en services/utils
- Formularios con validaciones reales
- Manejo cuidadoso de errores
- Nada de “todo en views.py” si ya se justifica dividir

FORMATO DE ENTREGA ESPERADO
1. Resumen inicial del repo encontrado
2. Plan de ejecución por lotes/commits
3. Implementación
4. Migraciones necesarias
5. Dependencias nuevas
6. Pasos para probar localmente
7. Riesgos, límites y próximos pasos
8. Crear pull request al terminar

REQUISITO FINAL
Lee y sigue además el archivo `instructions.txt` del repositorio o el archivo de instrucciones que se te indique. Ese archivo define el orden nocturno de ejecución y las reglas específicas de esta sesión.

Cuando termines:
- verifica el estado final del código
- resume lo implementado
- crea el pull request
- deja claro qué quedó listo y qué queda para una segunda fase
