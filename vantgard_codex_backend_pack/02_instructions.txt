instructions.txt — GUÍA OPERATIVA OPTIMIZADA PARA AGENTE NOCTURNO

Tu trabajo es AUTÓNOMO, PERO NO CAÓTICO.
Debes seguir exactamente este flujo.

==================================================
0. PRINCIPIOS DE EJECUCIÓN
==================================================
- Repo existente, no crear desde cero.
- Backend first.
- Cambios incrementales.
- Sin secretos reales.
- Sin sobreingeniería.
- Sin PR gigante desordenado.
- Si hay algo inconsistente, corrígelo con criterio técnico.
- Si una mejora estética no impacta funcionalidad, déjala para el final.

==================================================
1. INSPECCIÓN OBLIGATORIA DEL REPO
==================================================
Antes de modificar:
- inspecciona settings.py
- inspecciona urls.py del proyecto y de la app
- inspecciona models.py
- inspecciona admin.py
- inspecciona views.py
- inspecciona forms.py
- revisa si existen templates base, auth, home, account, producto, checkout
- revisa migraciones existentes
- detecta dependencias ya usadas

Debes producir un resumen técnico interno:
- qué existe
- qué falta
- qué está roto o incompleto
- qué puede reutilizarse

==================================================
2. ESTRATEGIA DE CAMBIO
==================================================
No mezcles todo.
Avanza en este orden:

PASO 1:
Estabilizar dominio y consistencia de modelos
- producto
- pedido
- detalle
- gift experience
- token de seguimiento
- pago o intentos de pago si hace falta
- entrega/logística si hace falta

PASO 2:
Endurecer formularios y validaciones

PASO 3:
Implementar servicios o utilidades para:
- generación de token
- QR
- cálculo de totales
- cálculo de despacho
- integración de pagos mock/sandbox

PASO 4:
Implementar vistas y urls del flujo de compra real
- catálogo/listado si hace falta mínimo soporte
- carrito
- checkout
- confirmación
- tracking por token

PASO 5:
Mejorar admin para operación real

PASO 6:
Ajustar templates mínimos necesarios
- solo lo necesario para soportar funcionalidad
- no perder tiempo en diseño decorativo excesivo

PASO 7:
Verificar migraciones, imports, enlaces y consistencia general

==================================================
3. CRITERIOS TÉCNICOS ESPECÍFICOS
==================================================
Modelos:
- campos coherentes
- nombres claros
- choices cuando tengan sentido
- validadores
- relaciones normales y escalables

Checkout:
- no confirmar pedido con stock insuficiente
- recalcular totales en servidor
- no confiar en datos del cliente
- separar subtotal, envío, descuentos y total

Pago:
- dejar arquitectura lista aunque sea mock
- registrar estado de pago
- registrar referencia/transacción si aplica
- permitir extender a Webpay, Mercado Pago u otro proveedor después

Tracking:
- token legible
- búsqueda por token
- no usar solo UUID crudo como experiencia final
- estados consistentes con tipo de entrega

QR:
- generar a partir de token o URL de seguimiento
- ligado al pedido
- reusable

Admin:
- filtros útiles
- búsqueda útil
- readonly apropiado
- acciones si ayudan
- stock crítico visible

==================================================
4. COSAS QUE NO DEBES HACER
==================================================
- No rehacer toda la app sin necesidad
- No mover media lógica a frontend
- No dejar lógica crítica solo en JS
- No meter credenciales
- No agregar librerías pesadas sin justificar
- No romper login, admin ni account existentes si ya funcionan
- No abrir demasiados frentes al mismo tiempo

==================================================
5. DEPENDENCIAS
==================================================
Solo agrega dependencias si son realmente útiles.
Ejemplos aceptables:
- qrcode
- Pillow
- requests
- python-dotenv

Si agregas una:
- actualiza requirements si existe
- documenta instalación
- explica para qué se usa

==================================================
6. CONTROL DE CALIDAD ANTES DE TERMINAR
==================================================
Debes revisar:
- modelos y migraciones consistentes
- imports válidos
- settings actualizados si aplica
- urls funcionando
- admin importando bien
- formularios válidos
- vistas sin errores obvios
- flujo de compra coherente
- tracking accesible
- QR enlazado correctamente
- PR listo para revisión humana

==================================================
7. FORMATO DE SALIDA FINAL
==================================================
Cuando termines el trabajo:
- crea un pull request
- incluye resumen por secciones:
  - contexto
  - cambios realizados
  - migraciones
  - dependencias
  - cómo probar
  - limitaciones
  - próximos pasos recomendados

==================================================
8. SI EL TIEMPO NO ALCANZA
==================================================
Prioriza dejar sólido:
1. modelos
2. checkout
3. tracking/token
4. QR
5. capa de pagos mock
6. admin

La consultoría sommelier / API externa es secundaria.
