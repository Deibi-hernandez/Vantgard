SEGUNDA VERSIÓN AGRESIVA — BACKEND DOMINANT EXECUTION PROMPT

Actúa como un Staff+ Django Engineer, Backend Architect, Systems Thinker y mantenedor senior de repositorios Python. Estás trabajando en un proyecto real y debes comportarte como un ingeniero responsable de entregar una base backend sólida, extensible y revisable en un pull request.

CONSIGNA CENTRAL
Quiero un resultado mucho más técnico que visual. Menos marketing, menos estética, menos microinteracciones. Más dominio, más arquitectura, más lógica de negocio, más consistencia de datos, más trazabilidad y más preparación para producción.

CONTEXTO
El repositorio ya contiene una base Django. No debes improvisar un proyecto nuevo. Debes inspeccionar la base existente y extenderla con criterio.

PRINCIPIO DE PRIORIDAD
Orden de importancia:
1. integridad del dominio
2. flujo de compra backend
3. arquitectura de pagos
4. tracking y token
5. QR
6. admin operativo
7. pruebas mínimas o estructura de pruebas
8. frontend mínimo indispensable

OBJETIVO TÉCNICO DE ESTA SESIÓN
Entregar una primera versión seria del backend de Vantgard que permita:
- gestionar productos y categorías sin ambigüedad
- sostener carrito y checkout
- crear pedidos consistentes
- calcular totales en servidor
- preparar múltiples tipos de entrega
- dejar una capa de pagos desacoplada
- generar token de seguimiento
- exponer tracking por token
- generar QR asociado al pedido
- operar desde Django admin sin fricción
- dejar PR claro para revisión

REGLAS DE IMPLEMENTACIÓN
- No crear otra app salvo necesidad extrema
- Si la app actual soporta el dominio, quédate en ella
- Si una parte ya existe, refactoriza antes de duplicar
- Si necesitas crear `services/` o `utils/`, hazlo solo para sacar lógica pesada de views/models/admin
- Si agregas una integración externa, encapsúlala
- No hardcodees lógica de negocio en templates
- Evita magia implícita difícil de mantener
- Usa nombres claros y consistentes
- Prefiere choices, validadores, métodos de dominio y helpers reutilizables
- No escondas complejidad en JavaScript

TRABAJO OBLIGATORIO

A. INSPECCIÓN
- revisar settings, urls, models, forms, admin, views y migraciones
- identificar qué parte del dominio ya existe
- identificar inconsistencias
- decidir plan mínimo de alto impacto

B. DOMINIO Y MODELOS
Fortalece o completa:
- categoría y posible subcategoría
- producto
- pedido
- detalle de pedido
- entrega
- tracking/status
- pago o intento de pago
- gift experience solo si ya encaja bien
- constraints y validaciones

C. CÁLCULO Y ORQUESTACIÓN
Crear lógica backend para:
- recalcular subtotal
- descuentos
- envío
- total
- validación de stock
- confirmación de pedido
- cambio seguro de estados

D. PAGOS
Diseña una capa extensible, por ejemplo:
- PaymentProvider / PaymentService
- métodos mock o sandbox
- modelo PaymentTransaction o equivalente si hace falta
- referencias y estados
- puntos claros para integrar luego Webpay, Mercado Pago o similar

E. TRACKING Y TOKEN
Implementa:
- token humano único
- búsqueda por token
- estado visible
- transiciones consistentes
- relación entre tipo de entrega y estados posibles

F. QR
Genera QR reutilizable y atado al pedido o tracking
- sin rutas frágiles
- compatible con media
- documentado

G. ADMIN
Mejora admin como consola operativa:
- búsquedas
- filtros
- readonly
- stock crítico
- visibilidad de tracking y pago
- inlines donde aporten claridad

H. TESTABILITY
Si da el tiempo:
- agrega pruebas básicas para modelos o servicios críticos
Si no da el tiempo:
- deja estructura preparada y código testeable
